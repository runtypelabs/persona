Durable Pause

Some pauses resume themselves. When the API emits an await event with an awaitReason (e.g. crawl_pending or durable_poll), the server — not the user — continues the stream. The widget shows a passive "working in the background" indicator and waits; it never renders a resume/input affordance.

Pause kind

Both are auto-resuming server pauses — only the indicator copy differs. The suppression of the resume affordance is driven by the presence of awaitReason, not by matching a specific value.

Notes

How it works

The unified await SSE event carries an optional awaitReason discriminator. When it is absent, the await is client-resolvable (a local tool / WebMCP call) and the widget renders the usual interactive affordance the client must fulfil. When it is a non-empty string, the pause is auto-resuming: the server (CrawlPollerDO for crawl, the wait-until Workflow for durable poll) resumes the same stream on its own, so the widget renders a passive variant: "pause" bubble and simply waits for subsequent frames.

Why no resume button

An auto-resuming pause has nothing for the client to fulfil — any resume action would have no payload to send. Showing a resume/input control there is a dead end, so the widget suppresses it and shows a non-interactive indicator instead. awaitReason is treated as UX context only, never a control signal: it decides how to render, never authorization, gating, or auto-resume logic.

Forward compatibility

Suppression keys on awaitReason being present and non-empty — not on an exact enum match — so a future durable-pause kind suppresses the affordance correctly without an SDK release. Copy can still special-case known kinds (and you can override it per-reason via config.copy.durablePauseLabels).

This demo

A customFetch replays a scripted stream: an intro reply, an await frame with the selected awaitReason, a few keep-alive pings (which must not settle the indicator), then the resumed answer and execution_complete — all on one stream. Watch the passive indicator appear, hold, and then disappear as the answer streams in. No backend or API key required.
