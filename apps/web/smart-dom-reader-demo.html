Featured drop shadow DOM

These cards render inside a shadow root. The default TreeWalker reader cannot see them; smart-dom-reader (full) can, so the assistant can too.

Notes

How it works

On every agent request the launcher widget runs a section-grouped context provider: it walks the shop's two <h2> sections and lists each product under its heading, crossing into the shadow root via host.shadowRoot for the Featured drop. Each product carries a product=<id> handle, and the result is merged into payload.context.pageContext.

Grouping by section (rather than the default reader's flat interactivity buckets) is what lets the assistant reliably answer "what's in the featured drop?": the heading and its products stay together instead of scattering across the prompt.

Use the Assistant's page reader toggle in the rail to switch the live method the launcher sends: Default, smart-dom-reader, or section-grouped, and watch the assistant gain (or lose) sight of the Featured drop and the ability to add it to the cart. The Compare readers buttons below it show the same readers' raw output side by side: smart-dom-reader (full mode) is the general-purpose library that pierces the shadow root the default collectEnrichedPageContext reader cannot reach.

Acting across the shadow boundary

The assistant doesn't just describe the shop: it can add products to the cart. It replies with a small JSON envelope ({"action": "add_to_cart", "product": "headphones", "text": "…"}); the host's handler resolves that product id to its Add-to-cart button and clicks it.

Persona's default click loop uses document.querySelector, which cannot pierce shadow roots, so the Featured-drop buttons would be unreachable. This demo's handler crosses the boundary with host.shadowRoot.querySelector('.buy-<id>'), so even the shadow-DOM products are clickable. Try "add the Studio Headphones".
