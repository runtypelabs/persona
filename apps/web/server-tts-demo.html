Server TTS (streaming)

Read assistant replies aloud with a hosted voice instead of the browser's built-in speech. A custom SpeechEngine streams PCM from the proxy's /api/tts route (OpenAI) into Persona's createPcmStreamPlayer(), so audio starts once the jitter buffer fills and pause/resume are real.

Voice & model

Both are read at speak time. Change them, then press “Read aloud” again. No widget rebuild.

How the engine plugs in

textToSpeech: { // no auto-speak; the button still uses this engine enabled: false, createEngine: () => new ServerTtsEngine({ endpoint: ttsUrl, }), }, messageActions: { showReadAloud: true },

The engine POSTs the message text to /api/tts and feeds the streamed 24 kHz PCM into createPcmStreamPlayer(), Persona's jitter-buffered AudioWorklet player, chunk-by-chunk. See src/server-tts-engine.ts and the route in examples/runtype-hono-proxy.

Notes

Requirements

This demo needs the proxy running (pnpm dev) with OPENAI_API_KEY set in examples/runtype-hono-proxy/.env (see .env.example). Without it, the reply still streams normally but “Read aloud” reports an error in the log.

Why a server engine?

The default BrowserSpeechEngine uses the Web Speech API: zero backend, but the voices are whatever the OS/browser ships and vary per device. A hosted engine gives one consistent, higher-quality voice everywhere, at the cost of a server call. textToSpeech.createEngine is the seam: return any object implementing SpeechEngine and the read-aloud button, the four-state icon, and the message:read-aloud event all work unchanged.

Why a jitter buffer

The proxy requests OpenAI's response_format: "pcm" (raw 24 kHz / 16-bit / mono) and pipes the body straight to the browser.

Hosted TTS arrives bursty: a brief blip, then a ~500–800ms generation stall, then a flood. Playing the first chunk immediately would starve and stutter.

createPcmStreamPlayer({ prebufferMs }) buffers a short waterline before the first sample and, on any underrun, goes silent and re-buffers instead of clicking. Pause/resume suspend the underlying AudioContext. Tune the latency↔smoothness trade-off with prebufferMs.

Swapping vendors

The browser contract is just “stream 24 kHz PCM”. server-tts-engine.ts does not care which service produced the bytes.

Runtype agents with voice configured can use POST /v1/client/agents/{agentId}/speak with the same client token as the widget. It streams PCM by default from the agent's configured voice provider, including ElevenLabs in production.

For a custom proxy like this demo, you can also swap the upstream fetch directly to ElevenLabs (/v1/text-to-speech/{voiceId}/stream?output_format=pcm_24000 with the xi-api-key header), Azure, Polly, or another provider.
