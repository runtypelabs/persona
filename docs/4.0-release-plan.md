## Interim release (2026-06-16): 3.36.0 unified-event consumer — PROTOTYPED

A non-breaking 3.x carrier that ships the **consumer half** of the API's new opt-in unified SSE
vocabulary (`POST /v1/dispatch?events=unified`), bridged back onto the widget's existing handlers.
The neutral unified vocab (`execution_*`/`turn_*`/`text_*`/`tool_*`, no `agent_`/`flow_` prefixes) is
the cleanest candidate yet for *the* open Persona protocol — this ships the consumer side at minor risk.

- **Spec:** `docs/specs/widget-unified-event-bridge.md` — full inverse mapping table, every field rename
  pinned to the handler that reads it; opt-in `events: 'unified'` (default `'legacy'`); transducer at
  `client.ts:3393` leaving the dispatch chain untouched; first-frame auto-detect.
- **Prototype (built + green off the spec):**
  - `packages/widget/src/utils/unified-event-bridge.ts` — stateful `UnifiedToLegacyBridge`
    (unified → legacy transducer).
  - `packages/widget/src/utils/__fixtures__/unified-translator.oracle.ts` — vendored copy of the api
    translator, used as the round-trip test oracle (fails loudly if api mapping ↔ widget bridge drift).
  - `packages/widget/src/utils/unified-event-bridge.test.ts` — **15 tests**: round-trip vs the oracle
    (text / thinking / tools / media / approval / artifacts / **WebMCP await**) + unit (kind routing,
    drops, error mapping, reflection fold). **tsc + eslint clean; full suite 1291 green.**
- **VERIFIED key fact (2026-06-16):** origin/main **3.35.0 already renders the local-tool/WebMCP pause**
  (`client.ts:2050`, `step_await{local_tool_required}`) **and** ships the **full resume path** in
  `session.ts` (`resumeFlow`/`webMcpToolCallId`/`toolOutputs`). The unified protocol collapses every
  await into one `await`, which the bridge routes through that existing path. → **the complete WebMCP
  pause → execute → /resume → continue round-trip works in 3.36 with no ported handler.**
- **Consequence for 4.0:** the held `.changeset/agent-await-pause.md` major is **vestigial under a
  unified-first strategy** — its +35 lines only render the *legacy* `agent_await` event name from
  non-unified agent dispatch. The bridge never sees `agent_await` (unified emits `await`). So that major
  stops being a 4.0 *gating* item; keep it only as long as legacy agent dispatch is supported.
- **Still to do to ship 3.36:** (a) wire the bridge into `client.ts` at the `seqBuffer.push` site (3393)
  behind the `events: 'unified'` config + first-frame auto-detect (bypass the reorder buffer — the
  unified stream is in-order); (b) add the `events` option to config + `InstallConfig` and append
  `?events=unified` to the dispatch + `/resume` URLs; (c) `minor` changeset; (d) branch off
  `origin/main` (NOT `persona-4.0`). Inert until the API honors `?events=unified` (same WS1 deploy gate,
  not a widget gate).

## Sequencing decision: API first

The breaking agent-dispatch behavior is gated on a **runtype-core (API)** change. We do that
**first**. The Persona-side branches are committed but **held** until the API change is live on the
upstream the proxy targets. Rationale: shipping the agent-path demos before the API supports them
reproduces the production failure (model fabricates tool calls as text; `toolCallsMade: 0`).

### 1. [DONE locally · VERIFIED e2e · needs deploy] Agent-dispatch `clientTools` + `agent_await`
- Implemented in `runtype-core` (local). Spec: `docs/specs/runtype-api-agent-clienttools-await.md`.
- **Change A:** thread dispatch `clientTools[]` into the agent's tool set (flow-path parity). ✅
- **Change B:** emit `agent_await` on a local-tool pause + resume into the agent loop. ✅
- **2026-06-15:** full Switchback WebMCP round-trip verified end-to-end against the local API on the
  `persona-4.0` integration branch (agent_await → widget runs page tool → `/resume` → agent continues).
- **Follow-up (correctness, recommended before 4.0):** the run's `executionId` switches from the
  agent's `exec_…` to a nested `virtual_…` at the first local-tool pause and stays there through
  `agent_complete` — the agent execution is abandoned at the pause and a virtual flow execution
  carries the rest. Works (renders fine) but breaks executionId correlation. Spec written:
  `docs/specs/runtype-api-agent-await-execution-continuity.md` (tightens Change B: keep one `exec_…`
  end to end; resume must re-enter the agent loop). Also verify `iteration` progression there.
- **Still needs to deploy** to the upstream the production proxy targets before releasing 4.0.
